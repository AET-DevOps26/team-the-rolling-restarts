# Helm Chart

This directory contains the Helm chart for the Kubernetes workloads in this repository. The chart renders the three application deployments, the three services, and the shared ingress resource from one values model.

The chart-local [Makefile](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/Makefile) wraps the common Helm commands so you can lint, render, install, upgrade, and remove the release without retyping the full command each time.

## What is in the chart

- [values.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/values.yaml) holds the shared defaults for all workloads, database config, and ingress/host settings.
- [values-prod.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/values-prod.yaml) overrides for production (e.g. `letsencrypt-prod`, 2 replicas).
- [secrets-values.example.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/secrets-values.example.yaml) template for the database credentials, the user-service JWT signing keys, and the service-to-service client secret — copy to `secrets-values.yaml` and fill in real values. **Never commit `secrets-values.yaml` to Git.**
- [image-values.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/image-values.example.yaml) is generated by the image upload workflow and only overrides image references.
- [templates/deployment.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/templates/deployment.yaml) renders the `web-client`, `api-gateway`, `user-service`, `content-service`, and `gen-ai` deployments.
- [templates/databases.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/templates/databases.yaml) renders the MongoDB deployment, service, and PVC (shared by user-service and content-service via separate databases).
- [templates/secrets.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/templates/secrets.yaml) renders the Kubernetes Secrets: database credentials, the `jwt-keys` signing pair, and the `service-credentials` client secret.
- [templates/service.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/templates/service.yaml) renders the matching services.
- [templates/ingress.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/templates/ingress.yaml) renders the external routes when `ingress.enabled` is true.

## Values Model

The chart is organized around a few shared knobs:

- `global.registry`, `global.tag`, and `global.pullPolicy` define the default container image behavior.
- `global.replicas`, `global.probes.path`, `global.resources`, and `global.service` provide cluster-wide defaults.
- Each workload block contains its own `name`, `imageName`, `port`, `env`, and probe timings.
- The `hosts` block feeds ingress routing for the web, API, and AI endpoints.
- `ingress.clusterIssuer` selects the cert-manager issuer (`letsencrypt-staging` by default).

If a workload-specific image is supplied through [image-values.yaml](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/image-values.example.yaml), the chart uses that image directly. Otherwise, it falls back to `global.registry/imageName:tag`.

## Environments

The base `values.yaml` targets development and uses `letsencrypt-staging` for TLS.

- **Dev / staging** (default): `make helm-deploy`
- **Production**: `make helm-deploy ENV=prod` — overlays `values-prod.yaml` which switches to `letsencrypt-prod` and sets 2 replicas.

## Database Credentials

Database credentials are stored in Kubernetes Secrets (not plain env vars), following the AET fair-use policy.

To set up credentials:

1. Copy the example file:

   ```bash
   cp secrets-values.example.yaml secrets-values.yaml
   ```

2. Edit `secrets-values.yaml` with your actual credentials.

3. Deploy with the secrets file:

   ```bash
   make helm-deploy SECRETS="-f secrets-values.yaml"
   ```

The `secrets-values.yaml` file is in `.gitignore` and must never be committed. For CI/CD, inject credentials via pipeline secrets or external secret management.

## Usage Guide

Render the chart locally without touching the cluster:

```bash
helm template newsgenai . -f values.yaml -f image-values.yaml
```

Lint the chart before deployment:

```bash
helm lint . -f values.yaml -f image-values.yaml
```

Install a new release:

```bash
helm install newsgenai . -f values.yaml -f image-values.yaml
```

Upgrade an existing release:

```bash
helm upgrade newsgenai . -f values.yaml -f image-values.yaml
```

Install or upgrade in a single command:

```bash
helm upgrade --install newsgenai . -f values.yaml -f image-values.yaml
```

Remove the release:

```bash
helm uninstall newsgenai
```

Inspect the final rendered manifests after changes:

```bash
helm template newsgenai . -f values.yaml -f image-values.yaml > /tmp/newsgenai-rendered.yaml
```

## Ingress Behavior

Ingress is controlled by `ingress.enabled`.

- When it is `true`, the chart renders a single Ingress with routes for the web app, API, and AI service.
- When it is `false`, the ingress manifest is omitted entirely.
- The ingress uses the configured `ingress.className`, the shared TLS secret, and the three hostnames in `hosts`.

## Makefile Targets

Use the chart-local [Makefile](https://github.com/AET-DevOps26/team-the-rolling-restarts/blob/main/infra/helm/Makefile) for the common Helm commands:

- `make help`
- `make helm-lint`
- `make helm-template`
- `make helm-install`
- `make helm-upgrade`
- `make helm-deploy`
- `make helm-destroy`

Append `ENV=prod` to any target to overlay production values (e.g. `make helm-deploy ENV=prod`).

## Workflow Integration

The image upload workflow writes a values file that only contains the workload image overrides. That means the deployment chart and the CI-generated image values can be combined safely without changing the chart structure.

The web-client image is built once and deployed identically everywhere — its API base URL is
read at server runtime via `API_BASE_URL` (`infra/helm/values.yaml`'s `webClient.env`, set to
`http://api-gateway:8080`, the bare in-namespace Service name), not baked in as a Docker build
argument. `web-client/src/lib/api/client.ts` (the only place that reads it) is `"server-only"`
and deliberately doesn't use the `NEXT_PUBLIC_` prefix, since that would get inlined into the
compiled bundle at build time and couldn't be overridden per deployment target afterward — which
is exactly the bug this convention exists to avoid (a stale build-time value would otherwise break
web-client's API calls in production).

## Notes

- The chart uses one template for all five workloads (web-client, api-gateway, user-service, content-service, gen-ai), so changing a shared value updates all rendered manifests consistently.
- The chart does not create namespaces.
- The ingress secret is `rolling-restarts-tls` by default.
