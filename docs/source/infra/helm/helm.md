# Helm Chart

This directory contains the Helm chart for the Kubernetes workloads in this repository. The chart renders the three application deployments, the three services, and the shared ingress resource from one values model.

The chart-local [Makefile](Makefile) wraps the common Helm commands so you can lint, render, install, upgrade, and remove the release without retyping the full command each time.

## What is in the chart

- [values.yaml](values.yaml) holds the shared defaults for all three workloads, plus the ingress and host settings.
- [image-values.yaml](image-values.yaml) is generated by the image upload workflow and only overrides image references.
- [templates/deployment.yaml](templates/deployment.yaml) renders the `web-client`, `api-gateway`, `user-service`, `content-service`, and `gen-ai` deployments.
- [templates/service.yaml](templates/service.yaml) renders the matching services.
- [templates/ingress.yaml](templates/ingress.yaml) renders the external routes when `ingress.enabled` is true.

## Values Model

The chart is organized around a few shared knobs:

- `global.registry`, `global.tag`, and `global.pullPolicy` define the default container image behavior.
- `global.replicas`, `global.probes.path`, `global.resources`, and `global.service` provide cluster-wide defaults.
- Each workload block contains its own `name`, `imageName`, `port`, `env`, and probe timings.
- The `hosts` block feeds ingress routing for the web, API, and AI endpoints.

If a workload-specific image is supplied through [image-values.yaml](image-values.yaml), the chart uses that image directly. Otherwise, it falls back to `global.registry/imageName:tag`.

## Usage Guide

Render the chart locally without touching the cluster:

```bash
helm template newsgenai . -f values.yaml -f image-values.yaml
```

Lint the chart before deployment:

```bash
helm lint . -f values.yaml -f image-values.yaml
```

Install a new release:

```bash
helm install newsgenai . -f values.yaml -f image-values.yaml
```

Upgrade an existing release:

```bash
helm upgrade newsgenai . -f values.yaml -f image-values.yaml
```

Install or upgrade in a single command:

```bash
helm upgrade --install newsgenai . -f values.yaml -f image-values.yaml
```

Remove the release:

```bash
helm uninstall newsgenai
```

Inspect the final rendered manifests after changes:

```bash
helm template newsgenai . -f values.yaml -f image-values.yaml > /tmp/newsgenai-rendered.yaml
```

## Ingress Behavior

Ingress is controlled by `ingress.enabled`.

- When it is `true`, the chart renders a single Ingress with routes for the web app, API, and AI service.
- When it is `false`, the ingress manifest is omitted entirely.
- The ingress uses the configured `ingress.className`, the shared TLS secret, and the three hostnames in `hosts`.

## Makefile Targets

Use the chart-local [Makefile](Makefile) for the common Helm commands:

- `make help`
- `make helm-lint`
- `make helm-template`
- `make helm-install`
- `make helm-upgrade`
- `make helm-deploy`
- `make helm-destroy`

## Workflow Integration

The image upload workflow writes a values file that only contains the workload image overrides. That means the deployment chart and the CI-generated image values can be combined safely without changing the chart structure.

The workflow also passes `NEXT_PUBLIC_API_BASE_URL` as a Docker build argument when building `web-client`. Next.js inlines `NEXT_PUBLIC_*` variables at build time, so the API URL must be set during the image build—not via Kubernetes deployment env vars at runtime.

## Notes

- The chart uses one template for all five workloads (web-client, api-gateway, user-service, content-service, gen-ai), so changing a shared value updates all rendered manifests consistently.
- The chart does not create namespaces.
- The ingress secret is `rolling-restarts-tls` by default.
